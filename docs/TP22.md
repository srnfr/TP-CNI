On utilise `cilium` en version **1.19.3** et la fonctionnalité de **chiffrement transparent L4** via [ztunnel](https://docs.cilium.io/en/stable/security/network/encryption-ztunnel/).

## But du TP

Cilium intègre **ztunnel**, un proxy per-nœud qui fournit un chiffrement mTLS transparent de couche 4 pour les communications pod-à-pod. Contrairement à SPIRE/SPIFFE (qui s'active par `CiliumNetworkPolicy`), ztunnel s'active par **label de namespace** et chiffre automatiquement tout le trafic TCP entre les pods enrôlés.

Architecture :
- Un **proxy ztunnel** tourne sur chaque nœud (DaemonSet dans `kube-system`)
- Les pods enrôlés ont leur trafic **redirigé via iptables** vers le proxy ztunnel local
- Le chiffrement mTLS utilise le port **15008** (protocole HBONE — HTTP/2 CONNECT over mTLS)
- L'authentification repose sur une **PKI dédiée** (clés bootstrap + CA) stockée dans un secret Kubernetes

Points clés :
- **Activation par namespace** — `kubectl label namespace <ns> io.cilium/mtls-enabled=true`
- **Transparent** — zéro modification applicative
- **TCP uniquement** — UDP non supporté
- **Incompatible avec ClusterMesh** — ne pas activer les deux simultanément
- **Les politiques réseau L4 ne fonctionnent pas** sur les pods enrôlés (trafic chiffré avant de quitter le pod)

## Etape 1 : Générer les secrets PKI

Ztunnel nécessite deux paires de clés/certificats, créées **avant** l'installation de Cilium :

- **Bootstrap** : sécurise la connexion entre ztunnel et le serveur xDS/certificats de Cilium
- **CA** : autorité de certification racine pour émettre les certificats clients mTLS éphémères

Générer les clés et créer le secret Kubernetes :

```shell
cat > /tmp/gen-ztunnel-secrets.sh << 'SCRIPT'
#!/usr/bin/env bash
# SPDX-License-Identifier: Apache-2.0
set -eu
cd /tmp

# == Bootstrap ==
openssl genrsa -out bootstrap-private.key 2048

cat > openssl-bootstrap.conf << 'EOF'
[ req ]
distinguished_name = req_distinguished_name
x509_extensions = v3_ca
prompt = no
[ req_distinguished_name ]
O = cluster.local
[ v3_ca ]
subjectKeyIdentifier = hash
authorityKeyIdentifier = keyid:always,issuer
basicConstraints = CA:FALSE
keyUsage = digitalSignature, keyEncipherment
extendedKeyUsage = serverAuth, clientAuth
subjectAltName = @alt_names
[alt_names]
DNS.1 = localhost
EOF

openssl req -x509 -new -nodes -key bootstrap-private.key \
  -sha256 -days 3650 -out bootstrap-root.crt \
  -config openssl-bootstrap.conf

# == CA ==
openssl genrsa -out ca-private.key 2048

cat > openssl-ca.conf << 'EOF'
[ req ]
distinguished_name = req_distinguished_name
x509_extensions = v3_ca
prompt = no
[ req_distinguished_name ]
O = cluster.local
[ v3_ca ]
subjectKeyIdentifier = hash
authorityKeyIdentifier = keyid:always,issuer
basicConstraints = critical, CA:true
keyUsage = critical, digitalSignature, cRLSign, keyCertSign
EOF

openssl req -x509 -new -nodes -key ca-private.key \
  -sha256 -days 3650 -out ca-root.crt \
  -config openssl-ca.conf

kubectl --namespace kube-system create secret generic cilium-ztunnel-secrets \
  --from-file=bootstrap-private.key=bootstrap-private.key \
  --from-file=bootstrap-root.crt=bootstrap-root.crt \
  --from-file=ca-private.key=ca-private.key \
  --from-file=ca-root.crt=ca-root.crt

echo "Secret cilium-ztunnel-secrets créé."
SCRIPT

chmod +x /tmp/gen-ztunnel-secrets.sh
bash /tmp/gen-ztunnel-secrets.sh
```

Vérifier que le secret est présent :

```shell
kubectl -n kube-system get secret cilium-ztunnel-secrets
```

## Etape 2 : Installer Cilium 1.19 avec ztunnel

Créer le fichier de valeurs Helm :

```shell
cat > /home/cilium_lab/basic/ztunnel-values.yaml << 'EOF'
kubeProxyReplacement: true
k8sServiceHost: basic-control-plane
k8sServicePort: 6443
image:
  pullPolicy: IfNotPresent
ipam:
  mode: kubernetes
encryption:
  enabled: true
  type: ztunnel
hubble:
  relay:
    enabled: true
  ui:
    enabled: true
EOF
```

Installer Cilium :

```shell
helm upgrade --install --namespace kube-system \
  --repo https://helm.cilium.io cilium cilium \
  --version 1.19.3 \
  --values /home/cilium_lab/basic/ztunnel-values.yaml
```

Attendre que Cilium soit prêt :

```shell
cilium status --wait
```

## Etape 3 : Vérifier ztunnel

Vérifier que ztunnel est activé dans la ConfigMap Cilium :

```shell
kubectl -n kube-system describe cm cilium-config | grep enable-ztunnel -A2
```

Vous devez voir `enable-ztunnel: "true"`.

Vérifier les pods ztunnel (un par nœud) :

```shell
kubectl -n kube-system get pods -o wide | grep ztunnel
```

Vous devez voir un pod `ztunnel-*` en `Running` sur chacun des 4 nœuds (control-plane + 3 workers).

## Etape 4 : Déployer les applications de test

Créer deux namespaces :

```shell
kubectl create namespace ns-a
kubectl create namespace ns-b
```

Déployer `traefik/whoami` dans `ns-a` (serveur) et un pod curl dans `ns-b` (client) :

```shell
kubectl -n ns-a create deployment app-a --image=traefik/whoami --port=80
kubectl -n ns-a expose deployment app-a --port=80 --target-port=80

kubectl -n ns-b run curl-tester --image=curlimages/curl \
  --restart=Never -- sleep 3600
kubectl wait -n ns-b pod/curl-tester --for=condition=Ready --timeout=60s
```

Vérifier que les pods sont Running :

```shell
kubectl get pods -n ns-a -o wide
kubectl get pods -n ns-b -o wide
```

Tester la connectivité de base **avant** enrôlement (trafic en clair) :

```shell
kubectl -n ns-b exec curl-tester -- \
  curl -s --max-time 5 http://app-a.ns-a.svc.cluster.local/
```

Cette requête doit **réussir** — ztunnel n'est pas encore actif pour ces namespaces.

## Etape 5 : Enrôler les namespaces dans ztunnel

L'enrôlement dans ztunnel s'active par un **label sur le namespace**. Dès qu'un namespace est enrôlé :
- Tous les pods existants sont redirigés vers ztunnel (règles iptables dans leurs netns)
- Les pods futurs sont automatiquement enrôlés au démarrage

Enrôler `ns-a` et `ns-b` :

```shell
kubectl label namespace ns-a io.cilium/mtls-enabled=true
kubectl label namespace ns-b io.cilium/mtls-enabled=true
```

Vérifier les namespaces enrôlés :

```shell
kubectl get namespaces -l io.cilium/mtls-enabled=true
```

Vérifier l'enrôlement effectif dans la StateDB de Cilium :

```shell
kubectl exec -n kube-system ds/cilium -- \
  cilium-dbg statedb dump | jq '.["mtls-enrolled-namespaces"]'
```

Vous devez voir `ns-a` et `ns-b` listés comme enrôlés.

## Etape 6 : Valider le chiffrement mTLS

### Test de connectivité

Générer du trafic depuis `ns-b` vers `ns-a` :

```shell
kubectl -n ns-b exec curl-tester -- \
  curl -s --max-time 10 http://app-a.ns-a.svc.cluster.local/
```

La connexion doit **réussir** — ztunnel chiffre le trafic de manière transparente.

### Vérifier le chiffrement par tcpdump

Identifier le nœud hébergeant le pod `app-a` :

```shell
APP_A_NODE=$(kubectl -n ns-a get pods -l app=app-a -o jsonpath='{.items[0].spec.nodeName}')
echo "app-a tourne sur : $APP_A_NODE"

CILIUM_ON_NODE=$(kubectl -n kube-system get pods -l k8s-app=cilium -o wide \
  | grep "$APP_A_NODE" | awk '{print $1}')
echo "Cilium pod : $CILIUM_ON_NODE"
```

Lancer tcpdump sur ce nœud pour observer le trafic chiffré sur le port HBONE (15008) :

```shell
kubectl -n kube-system exec $CILIUM_ON_NODE -c cilium-agent -- \
  bash -c "apt-get update -qq && apt-get install -y -qq tcpdump && tcpdump -i any port 15008 -c 20 -n" &
TCPDUMP_PID=$!

# Générer du trafic
kubectl -n ns-b exec curl-tester -- \
  curl -s --max-time 10 http://app-a.ns-a.svc.cluster.local/ > /dev/null

wait $TCPDUMP_PID 2>/dev/null || true
```

Vous devez voir des paquets sur le port **15008** (HBONE), ce qui confirme que le trafic passe par ztunnel et est chiffré en mTLS :

```
13:00:06.982499 IP 10.244.4.x.15008 > 10.244.1.x.xxxxx: ...
13:00:06.982536 IP 10.244.1.x.xxxxx > 10.244.4.x.15008: ...
```

### Observer avec Hubble

```shell
cilium hubble port-forward &
sleep 2
hubble observe --to-namespace ns-a --last 20
```

### Désenrôler un namespace (facultatif)

Pour désactiver ztunnel sur un namespace :

```shell
kubectl label namespace ns-b io.cilium/mtls-enabled-
```

Les règles iptables sont supprimées et ztunnel arrête de traiter le trafic de `ns-b`.

## Limitations

D'après la [documentation officielle](https://docs.cilium.io/en/stable/security/network/encryption-ztunnel/) :

| Limitation | Détail |
|-----------|--------|
| **TCP uniquement** | UDP et autres protocoles ne sont pas redirigés vers ztunnel |
| **Pas de ClusterMesh** | Incompatible — une erreur de validation est levée si les deux sont activés |
| **Pas de pod-level enrollment** | L'enrôlement se fait uniquement au niveau namespace |
| **Les deux endpoints doivent être enrôlés** | Pas de communication entre un pod enrôlé et un pod non-enrôlé |
| **Politiques L4 inopérantes** | Le trafic est chiffré avant de quitter le pod, les politiques L4 ne s'appliquent pas (sauf sur le port 15008 directement) |
| **Pods host-network exclus** | Les pods sans netns ne peuvent pas être enrôlés |
| **Requiert iptables** | Incompatible avec les environnements sans iptables |

## Cleanup

```shell
kubectl label namespace ns-a io.cilium/mtls-enabled- 2>/dev/null || true
kubectl label namespace ns-b io.cilium/mtls-enabled- 2>/dev/null || true
kubectl delete pod curl-tester -n ns-b --ignore-not-found
kubectl delete namespace ns-a ns-b
cd /home/cilium_lab/basic && ./clean-kind.sh
```
