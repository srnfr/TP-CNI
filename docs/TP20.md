## But du TP

Publier une application interne en HTTP puis HTTPS avec **Gateway API**.

Contrairement au TP15 (Ingress NGINX), ici nous utilisons directement la Gateway API avec l'implémentation Cilium.
Sur DigitalOcean, la `GatewayClass` Cilium est déjà installée.

## Etape 0 : Initialisation du numero de groupe

Positionner une variable qui contient votre numéro de groupe :

```shell
GRP=X
echo $GRP
```
Vous devez obtenir :
```
X
```

<ins>**Bien sûr utiliser la bonne valeur de X**</ins>

```shell
kubectl get gatewayclass

## Etape 3 : Publication HTTP avec Gateway API

Déployons une `Gateway` et une `HTTPRoute`.

```shell
cat > gateway-http.yaml << 'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: demo-gateway
  annotations:
    service.beta.kubernetes.io/do-loadbalancer-name: "lb-ga-grpX"
spec:
  gatewayClassName: cilium
  listeners:
    - name: http
      protocol: HTTP
      port: 80
      hostname: grpX.randco.eu
    - name: https
      protocol: HTTPS
      port: 443
      hostname: grpX.randco.eu
      tls:
        mode: Terminate
        certificateRefs:
          - kind: Secret
            name: demo-gateway-tls
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: demo-route
spec:
  parentRefs:
    - name: demo-gateway
  hostnames:
    - grpX.randco.eu
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: demo-service-v1
          port: 80
EOF
```
Appliquer en remplaçant `X` par votre groupe :

```shell
sed "s/X/$GRP/g" gateway-http.yaml | kubectl apply -f -
```

Vérifier le statut :

```shell
kubectl get gateway demo-gateway
kubectl get httproute demo-route -o wide
```

Récupérer l'adresse exposée  :

```shell
kubectl get gateway demo-gateway -o jsonpath='{.status.addresses[0].value}'
```
Tester en forçant l'IP :

```shell
curl -H "Host: grp${GRP}.randco.eu" http://<GATEWAY_IP>/
```

Si la résolution DNS est en place (attendre le GO de l'animateur), vous pouvez aussi tester :

```shell
curl http://grp${GRP}.randco.eu
```

## Etape 4 : Publication HTTPS avec Gateway API

Dans ce TP, on active HTTPS avec un certificat public Let’s Encrypt via cert-manager.

Prérequis importants :

1. `grp${GRP}.randco.eu` doit résoudre vers l'IP publique de la Gateway.
2. Le port 80 de la Gateway doit être accessible depuis Internet (challenge HTTP-01).

Installer cert-manager (avec support Gateway API) :

```shell
helm repo add jetstack https://charts.jetstack.io
helm repo update
helm upgrade --install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --create-namespace \
  --version v1.18.2 \
  --set crds.enabled=true \
  --set "extraArgs={--enable-gateway-api}"
```

Vérfier :
```shell
kubectl -n cert-manager rollout status deployment/cert-manager
kubectl -n cert-manager rollout status deployment/cert-manager-webhook
kubectl -n cert-manager rollout status deployment/cert-manager-cainjector
```

Créer un ClusterIssuer Let’s Encrypt  :

```shell
cat > clusterissuer-letsencrypt.yaml << 'EOF'
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    email: admin@null.com
    server: https://acme-v02.api.letsencrypt.org/directory
    privateKeySecretRef:
      name: letsencrypt-prod-account-key
    solvers:
      - http01:
          gatewayHTTPRoute:
            parentRefs:
              - name: demo-gateway
                namespace: default
                kind: Gateway
EOF
```

Appliquer le ClusterIssuer :

```shell
kubectl apply -f clusterissuer-letsencrypt.yaml
kubectl get clusterissuer letsencrypt-prod
```

Rappellons que nous avions déclarer la  `Gateway` pour écouter en HTTPS avec le secret `demo-gateway-tls` :

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: demo-gateway
spec:
  gatewayClassName: cilium
  listeners:
    [..]]
    - name: https
      protocol: HTTPS
      port: 443
      hostname: grpX.randco.eu
      tls:
        mode: Terminate
        certificateRefs:
          - kind: Secret
            name: demo-gateway-tls
```

Créeons donc l'objet `Certificate` qui contiendra le certificat Let’s Encrypt :

```shell
cat > certificate.yaml << 'EOF'
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: demo-gateway-cert
spec:
  secretName: demo-gateway-tls
  issuerRef:
    kind: ClusterIssuer
    name: letsencrypt-prod
  dnsNames:
    - grpX.randco.eu
EOF
```

```shell
sed "s/X/$GRP/g" certificate.yaml | kubectl apply -f -
```

Puis vérifier

```shell
kubectl describe certificate demo-gateway-cert
kubectl get secret demo-gateway-tls
kubectl get order,challenge
```

Quand le certificat est `Ready=True`, tester HTTPS :

```shell
curl "https://grp${GRP}.randco.eu"
```

## Etape 5 : Canary Deployment 50/50 avec Gateway API

Créer un second déploiement pour faire un canary deployment.

```shell
kubectl create deployment demo-v2 --image=traefik/whoami
kubectl expose deployment demo-v2 --name demo-service-v2 --port 80 --target-port 80
```

Vérifier que les deux services sont actifs :

```shell
kubectl get svc demo-service-v1 demo-service-v2
```

Mettre à jour la `HTTPRoute` pour router 50% vers chaque service. Créer une nouvelle HTTPRoute ou modifier l'existante :

```shell
cat > canary-route.yaml << 'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: canary-route
spec:
  parentRefs:
    - name: demo-gateway
  hostnames:
    - grpX.randco.eu
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: demo-service-v1
          port: 80
          weight: 50
        - name: demo-service-v2
          port: 80
          weight: 50
EOF
```

Appliquer la route de canary (remplacer `X`) :

```shell
sed "s/X/$GRP/g" canary-route.yaml | kubectl apply -f -
```

Vérifier la route :

```shell
kubectl get httproute canary-route
```

Tester le canary en effectuant plusieurs requêtes. Vous devriez obtenir alternativement des réponses du service v1 (nginx) et du service v2 (httpd) :

```shell
for i in {1..10}; do
  curl -s http://grp${GRP}.randco.eu/ | head -n 1
  echo "---"
done
```

Vous verrez alterner les noms de Pods (ex: `demo-v1-xxxxx` vs `demo-v2-xxxxx`), confirmant que le trafic est bien réparti 50/50.

Pour ajuster les poids (ex: 80/20) :

```shell
cat > canary-route.yaml << 'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: canary-route
spec:
  parentRefs:
    - name: demo-gateway
  hostnames:
    - grpX.randco.eu
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: demo-service-v1
          port: 80
          weight: 80
        - name: demo-service-v2
          port: 80
          weight: 20
EOF
```

```shell
sed "s/X/$GRP/g" canary-route.yaml | kubectl apply -f -
```

Vérifier la nouvelle répartition en effectuant plusieurs requêtes :

```shell
for i in {1..10}; do
  curl -s http://grp${GRP}.randco.eu/ | grep Hostname
done
```

Vous devriez voir environ 8 réponses `demo-v1-xxxxx` pour 2 réponses `demo-v2-xxxxx`.

Nettoyer ces routes HTTP :
```shell
kubectl delete httproute demo-route canary-route
```

## Etape 6 : Header-based Routing avec Gateway API

Le **header-based routing** permet d'aiguiller le trafic vers un backend différent selon la valeur d'un header HTTP, sans toucher aux poids. C'est utile pour tester une nouvelle version en production avec un public ciblé (ex: testeurs internes).

La règle est simple : si le header `X-Version: v2` est présent, le trafic va vers `demo-service-v2` ; sinon il va vers `demo-service-v1`.

Créer la `HTTPRoute` avec matching sur header :

```shell
cat > header-route.yaml << 'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: header-route
spec:
  parentRefs:
    - name: demo-gateway
  hostnames:
    - grpX.randco.eu
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
          headers:
            - name: X-Version
              value: v2
      backendRefs:
        - name: demo-service-v2
          port: 80
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: demo-service-v1
          port: 80
EOF
```

Appliquer (remplacer `X`) :

```shell
sed "s/grpX/grp$GRP/g" header-route.yaml | kubectl apply -f -
```

Vérifier :

```shell
kubectl get httproute header-route
```

Tester **sans** le header → réponse de `demo-service-v1` :

```shell
curl -s http://grp${GRP}.randco.eu/ | grep Hostname
```

Tester **avec** le header `X-Version: v2` → réponse de `demo-service-v2` :

```shell
curl -s -H "X-Version: v2" http://grp${GRP}.randco.eu/ | grep Hostname
```

Vous devriez voir des noms de pods `demo-v1-xxxxx` sans le header, et `demo-v2-xxxxx` avec le header.
```
